# RAG и поиск

## Источники

Фактически участвует только `memory_chunks` в SQLite. Потенциальные будущие источники:

- semantic memory из загруженных документов;
- episodic memory с подтвержденным опытом;
- 1С как оперативный источник, но не как память;
- шаблоны КП;
- пользовательский текущий контекст.

## Текущий full-text search

`MemoryStore.search_fts(query, namespace=None, limit=5)`:

- возвращает пустой список для пустого запроса;
- валидирует namespace, если он передан;
- валидирует `limit > 0`;
- ищет через `memory_chunks_fts MATCH ?`;
- присоединяет `memory_chunks`;
- возвращает словари с полями chunk и `score`;
- сортирует по `score`.

## Embeddings

Не реализованы. В README прямо указано, что embeddings не входят в текущий этап.

## Hybrid search

Не реализован. Возможная будущая схема:

1. Найти кандидатов через FTS5/BM25.
2. Найти кандидатов через vector search.
3. Слить списки.
4. Убрать дубли по `id`.
5. Переранжировать с учетом namespace, свежести, `priority`, `confidence`, источников.

## Rerank

Rerank пока отсутствует. В будущем rerank должен объяснять, почему выбран конкретный chunk, и возвращать source citation.

## Source citation

Текущий результат содержит `source`, `page`, `section`, `title`, поэтому базовые ссылки на источник возможны. Но нет единого формата citation и нет проверки, что источник действительно существует.

## Проблемы качества поиска

- FTS5 чувствителен к формулировкам запроса.
- Нет синонимов и нормализации терминов.
- Нет embeddings для смыслового поиска.
- Нет фильтрации по дате, типу документа и надежности кроме ручных полей.
- Нет тестов на качество выдачи.

## Что нужно протестировать

- Поиск по русскому тексту.
- Поиск с фильтром `namespace`.
- Пустой запрос.
- Неверный namespace.
- Ранжирование нескольких похожих chunk.
- Поиск по `title`, `text`, `source`.
- Поведение после удаления и обновления записей.
