# Журнал архитектурных решений

## Кратко

ADR в этой заметке объясняют, почему текущая карта проекта устроена именно так. Часть решений подтверждена кодом, часть является архитектурным правилом или гипотезой и явно помечена.

## ADR-001 SQLite как MVP-хранилище памяти

- Дата: дата неизвестна.
- Статус: принято фактически.
- Контекст: проекту нужен локальный Memory Store без отдельной инфраструктуры.
- Решение: использовать SQLite и FTS5.
- Альтернативы: PostgreSQL, Qdrant, файловое хранилище.
- Последствия: старт простой, но нужны миграции схемы и будущая стратегия vector search.
- Связанные файлы: [schema.py](../../linehelper/memory/schema.py), [memory_store.py](../../linehelper/memory/memory_store.py).
- Связанные заметки: [[05_Memory_System]], [[07_RAG_Search]], [[14_Tech_Debt#TD-006 Нет миграций схемы]].

## ADR-002 Разделение semantic и episodic memory

- Дата: дата неизвестна.
- Статус: принято фактически.
- Контекст: нужно отделить стабильные знания от подтвержденного практического опыта.
- Решение: использовать namespace `semantic` и `episodic`.
- Альтернативы: одна таблица без namespace; разные таблицы для каждого типа памяти.
- Последствия: проще фильтровать поиск, но нужна application-level policy для того, что можно сохранять.
- Связанные файлы: [memory_store.py](../../linehelper/memory/memory_store.py).
- Связанные заметки: [[05_Memory_System]], [[10_Commercial_Proposal_Flow]], [[14_Tech_Debt#TD-005 Нет политики безопасного сохранения памяти]].

## ADR-003 1С как источник актуальных данных а не память

- Дата: дата неизвестна.
- Статус: планируется как архитектурное правило, реализации нет.
- Контекст: оперативные данные устаревают и должны запрашиваться из источника истины.
- Решение: не сохранять данные 1С в semantic или episodic memory автоматически.
- Альтернативы: кешировать данные 1С в памяти; делать регулярные выгрузки.
- Последствия: нужен надежный интеграционный слой и правила кеширования, если кеш появится.
- Связанные файлы: не подтверждено текущей структурой проекта.
- Связанные заметки: [[06_1C_Integration]], [[10_Commercial_Proposal_Flow]], [[14_Tech_Debt#TD-004 Нет контракта интеграции с 1С]].

## ADR-004 TTL для episodic memory

- Дата: дата неизвестна.
- Статус: принято фактически.
- Контекст: практический опыт КП и сделок может устаревать.
- Решение: `save_experience()` задает `expires_at`, а `expire_old_episodes()` удаляет устаревшие episodic-записи.
- Альтернативы: ручное удаление; постоянное хранение; отдельная таблица истечения.
- Последствия: важен процесс пересмотра или продления ценных эпизодов.
- Связанные файлы: [memory_store.py](../../linehelper/memory/memory_store.py), [test_memory_store.py](../../scripts/tests/test_memory_store.py).
- Связанные заметки: [[05_Memory_System]], [[10_Commercial_Proposal_Flow]], [[11_Testing_Map]].

## ADR-005 Возможная миграция на vector DB

- Дата: дата неизвестна.
- Статус: гипотеза.
- Контекст: FTS5 не решает смысловой поиск и не заменяет embeddings.
- Решение: после MVP рассмотреть Qdrant или другую vector DB, сохранив SQLite как metadata store.
- Альтернативы: SQLite extensions; PostgreSQL pgvector; внешний поисковый сервис.
- Последствия: уже сейчас полезно сохранять стабильные ids, source и metadata.
- Связанные файлы: не подтверждено текущей структурой проекта.
- Связанные заметки: [[07_RAG_Search]], [[15_Roadmap]], [[14_Tech_Debt#TD-002 Поиск только FTS5 без embeddings]].
